# ADR-006: Public Media Serving for Animal Images and Videos

## Status

Accepted

## Context

The public API exposes adoptable animals to external consumers such as shelter websites. Those websites need to display animal images and videos directly — using them in `<img src>` or `<video src>` attributes without any client-side token handling.

The existing media system uses signed temporary URLs (1-hour expiration) generated by a custom `MediaUrlGenerator` and served through the authenticated `media.serve` route. These are unsuitable for public linking because:

- URLs expire, requiring consumers to periodically re-fetch the API just to refresh media links.
- Signed URLs cannot be embedded directly in HTML on third-party websites.

Animal images have no meaningful privacy concern. Person images and other model media (pre-inspection forms, etc.) do, and must remain inaccessible to the public.

## Decision

Introduce a dedicated unauthenticated route `GET /api/media/{mediaUuid}` (`api.media.serve`) served by `PublicMediaController`. The URL is permanent because the UUID is assigned on upload and never changes.

Two guards are enforced before serving a file:

1. **Model type check**: `$media->model_type === Animal::class` — ensures Person media and any other model's media is never exposed, even if the caller knows the UUID.
2. **Publication check**: `$media->model->do_publish === true` — ensures media for unpublished (adopted, removed) animals stops being served.

Both checks return `404` to avoid leaking that a UUID exists on a non-animal model.

Responses include `Cache-Control: public, max-age=604800` (one week). A one-year max-age would also be technically safe (content at a UUID is immutable — new uploads get new UUIDs), but one week limits the window where a CDN or browser cache might continue serving media after an animal is unpublished.

`PublicAnimalResource` includes a `media` array with `thumbnail`, `url` (preview), and `full` permanent URLs for images, and `url`/`full` for videos (no conversions for video).

## Consequences

### Positive

- Stable, directly-linkable URLs — no periodic re-fetching of API responses to refresh media links.
- Access control is retained: unpublished animals' media returns 404.
- URLs are application-controlled (route-based), not tied to the internal storage path structure. Migrating from local disk to S3 or reorganising storage does not break public URLs.
- Works as a CDN origin without additional configuration.

### Negative

- After an animal is unpublished, its media may continue to be served from browser or CDN caches for up to one week.
- The public endpoint requires a database lookup per request (media UUID → animal → do_publish). At typical shelter scale this is negligible, but a CDN in front of the API eliminates the concern entirely.

## Alternatives Considered

**Move animal pictures to the public disk** — Files would be served as static assets with direct filesystem URLs. Rejected because: no per-request access control (all uploaded images permanently accessible regardless of publication status), URLs are tied to Spatie's internal path structure, and migrating existing media on the local disk would require a storage migration.

**Long-lived signed URLs** — Use `getTemporaryUrl(now()->addYears(10))` in `PublicAnimalResource`. Rejected because URLs still expire (rotation complexity increases), and the signed URL infrastructure was designed for authenticated internal use.
