# AI GitHub Automation (Claude)

How Claude is wired into this repo's GitHub issues via `anthropics/claude-code-action`, what it can and can't do, and what's planned next.

For *why* this approach (vs. a custom-built agent, vs. a managed SaaS) was chosen, see [ADR-011](ADRs/ADR-011-claude-code-github-automation.md).

> [!NOTE]
> This is a separate, narrower initiative from [docs/ai-development-process.md](ai-development-process.md), which describes the broader human-driven loop using CodeRabbit and interactive Claude Code sessions. This doc covers a growing, fully automated slice of that loop (issue planning and implementation today; PR review is planned — see Roadmap below). Whether CodeRabbit's issue-planning role eventually moves onto this automation, or the two continue to coexist, hasn't been decided yet.

## What's implemented

**One command today: `/refine`.** Comment `@claude refine` at the start of a comment on an open issue (see [CONTRIBUTING.md](../CONTRIBUTING.md#using-claude-on-issues) for the user-facing how-to). It writes or updates a single tracked implementation plan comment on that issue:

- **First run** (no existing plan comment): writes an initial plan from the issue title/body and any comments already present.
- **Every following run**: rewrites that same comment in place, treating every comment posted since as feedback to incorporate.

The plan comment always starts with an invisible `<!-- claude-plan -->` marker, which is how the skill finds "its" comment again on the next run.

### Comment structure

Defined in [`.claude/skills/refine/SKILL.md`](../.claude/skills/refine/SKILL.md):

1. **Summary** — 1-3 sentences, high level, benefit-focused, no implementation detail.
2. **Status** — ✅/⚠️ derived deterministically: `Ready to implement` only if Confidence is High *and* there are no open questions left; every other combination is `Currently being refined`. Not GitHub's native `[!TIP]`/`[!WARNING]` alert syntax — that syntax's header text is one of five fixed keywords that can't be relabeled to "Status".
3. **Plan Metadata** — one line: Confidence and Complexity (each ✅/⚠️/🚨-style, deterministic per the skill's mapping), and Scope. Scope uses this repo's own domain vocabulary (`people`, `animals`, `adoption`, `inspection`, `contract`, `transport`, `form-builder`, `organizations`, `public-api`, `auth`, `infrastructure`, `docs`) sourced from `api/CONTEXT.md` / `frontend/CONTEXT.md`, not generic technical layers (backend/frontend/db/...) — chosen so a plan's scope is filterable/greppable by the part of the product it touches, not by which files change.
4. **Functional Plan** — concrete problem + behavior description, with a collapsible for edge cases / extended decisions.
5. **Technical Plan** — high-level approach, with a collapsible for file/module-level detail (written for a developer and for a future implementation agent, not just a human skimming).
6. **Resolved Decisions** — always-visible heading with an always-present collapsible underneath (defaults to "No decisions made yet.") — the answers to questions asked while refining.

No external images anywhere in the template (shields.io-style badges were considered and rejected — see Decisions below) — everything is plain markdown and native emoji.

### Workflow

[`.github/workflows/claude-refine.yml`](../.github/workflows/claude-refine.yml):

- Triggers on `issue_comment: created`, gated to issues (not PRs).
- Runs in **automation ("agent") mode** — the action is given an explicit `prompt` input rather than relying on tag/interactive mode, so Claude is handed a fixed instruction rather than left to infer intent from the comment text.
- Tool access is allowlisted to exactly what the skill needs: `Read`, `Glob`, `Grep` (read-only repo inspection, for the Technical Plan) and two narrow Bash patterns, `Bash(gh api:*)` / `Bash(gh issue comment:*)`.

**One more command: `/implement`.** Comment `@claude implement` on an issue (once its `/refine` plan says `Status: Ready to implement`) to have Claude write the code and open a PR against `development`; comment `@claude implement` on the resulting PR to have it act on review feedback or direct change requests and push more commits. One merged skill/workflow handles both — see [`.claude/skills/implement/SKILL.md`](../.claude/skills/implement/SKILL.md) and [`.github/workflows/claude-implement.yml`](../.github/workflows/claude-implement.yml):

- **Build mode** (triggered from an issue, no open PR yet): reads the `/refine` plan comment as its spec. Refuses (posts an explanatory issue comment instead) if the plan isn't `Status: Ready to implement`, or if a PR already exists for that issue.
- **Iterate mode** (triggered from an open PR): reads PR conversation comments and inline review feedback since the last update, adjusts the code, pushes more commits to the same branch. **Feedback is evaluated, not blindly applied** — for every distinct comment, Claude decides to implement it, decline it with reasoning, or ask for clarification, and a deterministic workflow step replies individually to each inline review comment with that verdict (threaded under the original comment, via GitHub's reply-to-review-comment API). Conversation-level comments and review bodies have no threading primitive, so those get addressed in the aggregate PR comment instead. Declining feedback with a clear reason is treated as just as valid an outcome as implementing it.
- Before Claude runs, the workflow sets up a full CI-parity environment (PHP 8.3 + MySQL 8.4 service container + Composer install + migrate, Node 24 + npm install) mirroring `.github/workflows/ci.yml` exactly, so the checks Claude runs inside its turn (`pint --test`, `php artisan test`, `npm run lint`/`format`/`test`/`build`) actually predict what CI will do on the resulting PR.
- Claude never talks to GitHub directly in this command — not even to post a comment. It ends its turn with structured JSON output (`--json-schema`, `mode`/`checks_passed`/`pr_title`/`pr_body`/`what_changed`/`blocked_reason`), and plain deterministic workflow steps afterward create the PR, post the iterate summary, toggle it ready, or post the blocked explanation. If checks didn't fully pass, the PR is still opened, as a GitHub draft, with the failures explained in the body.
- **Claude commits its own work incrementally, via a real-time API commit tool, not raw `git`.** `use_commit_signing: true` on the action step installs a `commit_files`/`delete_files` MCP tool that pushes commits straight to GitHub's Git Data API as Claude calls them — no local `git commit`/`git push` involved at all. The skill instructs Claude to commit after each logical unit of work, and once more immediately before producing its final output, rather than saving everything for one commit at the end. The workflow computes the branch name itself (a deterministic pre-Claude step) rather than trusting the action's own branch-naming inputs, since those only apply to a code path agent mode doesn't use — see the changelog below for how this was actually found.
- **If Claude's own turn fails outright** (API error, or running out of `--max-turns` before finishing), a narrower recovery step catches whatever's still sitting uncommitted in the checkout's working tree — i.e. edits made via `Edit`/`Write` that were never wrapped in a `commit_files` call before the run died. It commits anything dirty itself (as `claude[bot]`), then pushes `HEAD` to the same deterministically-computed branch, if it differs from a SHA captured before Claude ran. In build mode, if that push happened, it's then opened as a draft PR explaining the run didn't finish. If there was truly nothing left to recover (or this was an iterate-mode failure, where recovery already pushed straight onto the existing PR), a plain comment linking to the failed run is posted instead.

## Security model

- **Author gating**: only comments from users with `OWNER`, `MEMBER`, or `COLLABORATOR` association on the repo can trigger the job. This is a public repo; random commenters can't invoke it or spend API budget.
- **Exact-prefix trigger matching**: the job's `if:` uses `startsWith(comment.body, '@claude refine')`, not `contains(...)`. A comment that merely *mentions* `@claude` or `refine` in passing (e.g. "hey @claude, how does refine work?") does not trigger the job.
- **No slash in the human-facing trigger**: the GitHub-facing convention is `@claude <command>` with no `/`, since slash commands are a Claude Code/terminal convention rather than a typical GitHub-bot one. The literal `/refine` is still used internally, in the `prompt:` string handed to the Claude Code CLI — that's the correct layer for a slash command, since it invokes the skill of that exact name deterministically.
- **Deterministic command routing**: the workflow's gate decides which command runs and hands Claude a hardcoded prompt; Claude never chooses between skills itself. Each future command should follow the same pattern — its own job, its own exact-match gate, its own hardcoded prompt.
- **Least-privilege tool access**: no general `Bash`, no `Write`/`Edit` tool. Claude can read the repo (to name real file paths in the Technical Plan) and talk to GitHub only through the two allowlisted `gh` command shapes — it cannot modify the working tree or run arbitrary shell commands.
- **Least-privilege workflow permissions**: `contents: read`, `issues: write`, `pull-requests: read`, `id-token: write`. No `contents: write` — this job cannot push commits or open PRs.
- **Ephemeral OIDC token**: `id-token: write` lets the action mint a short-lived GitHub App token per run via OIDC instead of using a long-lived PAT.
- **Per-command permission isolation**: each command lives in its own workflow/job rather than a shared dispatcher, specifically so permissions can differ per command. `/implement` is the concrete example: it needs `contents: write` and `pull-requests: write` (to push commits and manage PRs) that `/refine` deliberately doesn't have — see ADR-011 for the rationale.
- **`/implement` goes further than `/refine` on tool access: almost zero GitHub write capability for Claude, with one deliberate, narrow exception.** `/refine` grants `Bash(gh issue comment:*)` because posting the comment *is* its whole mechanism. `/implement` needs to create PRs, comment, and toggle draft state — for comments and ready-toggling, every one of those actions is driven by Claude's `structured_output` (a `--json-schema` result) and executed by a plain, non-LLM workflow step afterward, using `github.token`. **PR creation itself is the one exception**: Claude calls `gh pr create` directly, as the very last thing it does in build mode (SKILL.md Step 3.5), using its own live GitHub App token rather than routing through a deterministic step — see the changelog below for why (in short: that token is dead by the time any later workflow step runs, and the fallback of using `github.token` there instead is blocked outright unless the repo's "Actions can create PRs" setting is enabled, which this repo deliberately leaves off). The practical risk delta is small — the deterministic-step version already trusted Claude's `pr_title`/`pr_body` content verbatim; this only changes *which* credential submits that same content. Claude's own allowlist is `Read,Glob,Grep,Edit,Write`, the `commit_files`/`delete_files` API-commit tool (see above), four read-only-or-narrowly-scoped Bash patterns (`gh api`, `gh pr view`, `gh pr diff`, `gh pr create`), and the exact CI check commands (`pint --test`, `php artisan test`, `npm run lint`/`format`/`test`/`build`) — no install commands (enforces `AGENTS.md`'s "no packages without consent" rule at the permission level, since there's no human mid-run to grant it), no raw `git` ever, and no `gh pr comment`/`gh pr ready`/`gh pr merge`/`gh issue comment` — those still go through the JSON only.
- **Untrusted-checkout / TOCTOU protection on `/implement`'s iterate mode.** Flagged by GitHub Advanced Security (CodeQL) after this workflow shipped: the author-association check only validates who *commented*, not what code is actually checked out and executed — and iterate mode was resolving the PR's branch by *name*, which is re-resolvable (new commits could land between the check and the checkout) and unrestricted (it would happily build/test any open PR a collaborator pointed it at). This is a privileged job (`contents: write`, `id-token: write`, the Claude secret) running `composer install`/`npm ci`/full test suites — i.e., executing whatever code is on that branch — so this mattered. Fixed in `claude-implement.yml`'s ref-resolution step: resolve the head **commit SHA** (`headRefOid`), not just the branch name, and check that SHA out (`actions/checkout`'s `ref:` pinned to a SHA can't be moved out from under it); reject the PR outright if it's from a fork (`isCrossRepository`); reject it if the branch isn't under `claude/implement/*` — this job now only ever executes code that originated from `/implement`'s own build step (or a trusted collaborator's follow-up commits on that same branch), never an arbitrary PR.

## Decisions made along the way

- **One merged skill, not two.** `plan-issue` and `refine-plan` were originally separate skills, but the underlying trigger (any `@claude` comment) was already generic, so Claude had to guess which one applied. Merged into a single `refine` skill that checks for the marker comment itself (POST if absent, PATCH if present) — deterministic instead of guessed.
- **Command naming**: renamed to `/refine` to match already-established internal terminology.
- **Comment body is composed inline, never written to a file.** The skill originally wrote a `plan.md` file before posting/patching it — but no `Write` tool is granted, so that step silently failed (see the permission-debugging note below). Fixed by composing the body as a heredoc passed directly into the `gh` command.
- **`gh api`, not `gh issue view`**, for reading the issue — both are equally read-only, but standardizing on one command keeps the tool allowlist to a single pattern instead of two that do the same thing.
- **Status derived by a literal rule**, not left to per-run judgment, so it stays trustworthy across refinements.
- **Scope vocabulary replaced, not supplemented.** Considered adding a second "Feature Area" field alongside a technical-layer Scope field; decided to replace Scope's values instead — a technical-layer tag is usually redundant once the domain area is known, and duplicating it would just add table noise.
- **No external images.** shields.io-badge-style color coding was proposed for Confidence/Complexity, then rejected in favor of native emoji — avoids any dependency on a third-party image service for something that renders inside every issue.
- **No GitHub alert syntax for Status.** `> [!TIP]` / `> [!WARNING]` looked good but their header text ("Tip", "Warning", ...) is one of five fixed keywords GitHub doesn't let you relabel — using it left "Status:" awkwardly duplicated under a generic "Tip" heading. Replaced with a plain emoji-prefixed line instead.
- **Debug/telemetry footer (turns used, token cost, refinement count) considered and deferred.** Turn count and cost are only known to the *harness* after a run completes (surfaced via the action's `execution_file` output), not to Claude while it's running — Claude cannot self-report these. A refinement counter, by contrast, *could* be self-maintained by Claude (it already re-reads the old comment each run). Decided to defer both: GitHub's own comment edit history already answers "how often was this changed," and wiring up the cost/turn footer needs an extra workflow step that isn't worth building yet. See Roadmap.
- **`/implement` merges build and iterate into one skill, not two**, mirroring the `plan-issue`/`refine-plan` merge. Justified by more than symmetry: `claude-code-action`'s own "Smart Branch Handling" already creates a new branch when triggered from an issue and pushes straight to the existing branch when triggered from an open PR, so the git mechanics are already unified by the harness — the two modes only differ in *what Claude reads as its spec* (the `/refine` plan comment vs. PR feedback) and *what a deterministic step does afterward* (create a PR vs. post a comment).
- **PR creation is a deterministic workflow step, not a Claude tool call.** Confirmed the action does not create PRs automatically — by design, per its own `docs/security.md`, the default is a link for a human to click, as an oversight safeguard. Rather than allowlisting `gh pr create` for Claude (one more thing an LLM tool call could get wrong, forget, or mistime), Claude reports a `pr_title`/`pr_body`/`checks_passed` via structured output and a plain workflow step creates the PR from that plus the action's own `branch_name` output. Extended to *every* GitHub write `/implement` needs (comments, ready-toggling too), not just PR creation.
- **Checks failing still produces a PR — as a draft**, rather than either blocking silently or holding the gate open indefinitely. A visible, inspectable draft (with the failures explained) beats no artifact at all; the check-and-fix loop is bounded (3 cycles) specifically so a stuck loop degrades to this instead of running forever.
- **No package installs, ever, for `/implement`.** `AGENTS.md` requires human consent before adding a dependency; since there's no human mid-run to ask, the only safe answer is to never grant the tool access that would let it happen — `composer install`/`npm ci` are run as pre-Claude workflow steps (so Claude never needs them either), and `composer require`/`npm install <pkg>` are never allowlisted at all. A belt-and-suspenders diff check on lockfiles in the PR-creation step forces `--draft` if one slips through anyway.
- **Iterate mode evaluates feedback instead of complying with all of it.** The initial design just had Claude implement whatever PR comments asked for. Extended `feedback_responses` in the structured output so each piece of feedback gets an explicit verdict (implemented/declined/clarification needed) with reasoning, replied to individually on the original comment thread by a deterministic step — kept inside the existing "Claude has zero GitHub write capability" pattern rather than granting `gh pr comment` directly.

### Permission debugging notes (for future commands hitting the same issues)

Two real bugs surfaced only once the workflow was actually triggered against a live issue — both left `permission_denials_count > 0` in the run's result summary:

1. **Missing repo-read tools.** The Technical Plan step asks Claude to "inspect the repo to name real paths," but `Read`/`Glob`/`Grep` weren't allowlisted. Fixed by adding them.
2. **Missing `Write` tool for the file-based comment-body approach.** Fixed by dropping the file entirely (see above) rather than granting `Write`.
3. **Claude spawned an async background `Explore` subagent to research the codebase, then called `ScheduleWakeup` to defer and resume later — both features of the interactive Claude Code product, meaningless in this context.** `claude-code-action` runs as a single one-shot process: the moment Claude's turn returns, the job writes its outputs and exits — there's no persistent session for a wakeup to fire into, and nothing ever polls a background agent. Claude even canceled its own scheduled wakeup a few seconds later and ended the turn saying it would "continue once it reports back" — there was no mechanism for that to happen. The run reported `is_error: false` / `subtype: "success"` / zero `permission_denials`, completely invisible as a failure, since it technically finished cleanly without ever reaching the skill's later steps (writing/posting the plan comment). Worse: `--allowedTools` did **not** block `Skill`, `Agent`, or `ScheduleWakeup` at all — they aren't gated by it. Fixed by adding `--disallowedTools` (an explicit blocklist, not an allowlist) covering the whole async/session-persistence toolset (`Agent`, `Task`, `ScheduleWakeup`, `Cron*`, `RemoteTrigger`, `PushNotification`, `SendMessage`, `Monitor`, `Task*`, `EnterWorktree`/`ExitWorktree`, `DesignSync`) on both `claude-refine.yml` and `claude-implement.yml` — neither skill ever needed these; codebase inspection is meant to happen directly via `Read`/`Glob`/`Grep`.

A run can report `"is_error": false"` / `"subtype": "success"` in its result summary while still having done nothing useful — that field means the Claude Code session didn't crash, not that the task succeeded. When debugging a "ran but did nothing" case, `show_full_output: true` on the action step (currently enabled in `claude-refine.yml`) reveals the actual tool-call transcript instead of just the summary JSON — worth checking whether it should be turned back off once the workflow has proven stable for a while, to keep run logs shorter.

### `/implement` — open risks, not yet live-tested

Built the same way `/refine` was designed, but not yet exercised against a real issue, so treat the following as things to watch for rather than settled behavior (mirrors how `/refine`'s actual permission bugs were only found by triggering it for real):

- **Bash allowlist pattern matching for colon-containing commands** (e.g. does `Bash(npm run lint:*)` match `npm run lint:fix`?) — designed around by keeping colon-fragile commands (`key:generate`, `migrate`) out of Claude's allowlist entirely and running them as pre-Claude workflow steps instead, but unconfirmed.
- ~~`github.event.issue.pull_request != null` as an `if:`/`env:` expression~~ — confirmed live on the first real trigger (an issue-mode run): the `!= null` form correctly evaluated to false and skipped the ref-resolution step, same as `/refine`'s negated-truthy form.
- **The duplicate-build guard's search query** (`gh api search/issues ... head:claude/implement/issue-{n}`) — not yet smoke-tested against a real open PR.
- **Path-scoped `Edit`/`Write` deny** for `.github/workflows/**`/`.claude/**` isn't implemented — no confirmed syntax for it exists yet, so that protection is prompt-level only (plus the lockfile-diff guard) for now.
- **`POST /pulls/{pr}/comments/{comment_id}/replies`** (used to reply to an individual review comment thread) — a documented, stable GitHub REST endpoint, but not yet exercised against this repo's actual token/permission setup; if it 403s, the fallback is folding that feedback item into the aggregate `what_changed` comment instead (same as `comment_type: "general"` already does).
- **`gh pr view --json headRefOid,isCrossRepository,author`** — documented `gh` CLI fields, but not yet run live in this workflow; if a field name has changed, the ref-resolution step fails closed (`exit 1` before checkout), which is the safe direction for this to break in.
- **`use_commit_signing` + `CLAUDE_BRANCH` + `mcp__github_file_ops__*` in the allowlist — not yet live-tested as a combination.** Confirmed by reading the action's source (see changelog below) but not yet exercised end-to-end: whether `commit_files` actually gets called by Claude as instructed, whether `CLAUDE_BRANCH` correctly flows through a composite action's env to make `steps.claude.outputs.branch_name` match the workflow's own `steps.branch.outputs.name`, and whether commit-signing's real-time API pushes interact cleanly with the local-working-tree recovery step's own `git push` on a failure (a non-fast-forward push there is treated as "already covered by commit_files," not retried — see the recovery step's comment in the workflow).

A follow-up hardening pass (prompted by a GitHub Advanced Security / CodeQL finding on the untrusted-checkout fix above) closed a few more gaps: the PR-creation and iterate-ready-toggle lockfile guards now use `gh api compare` instead of raw `git diff`, which let `persist-credentials: false` be set on the checkout step (nothing needed the token it would otherwise leave in local git config); and the job has a `concurrency` group keyed on the issue/PR number so overlapping `@claude implement` comments serialize instead of racing. (That pass also added a check that the PR was authored by `claude[bot]`, on top of the `claude/implement/*` branch-prefix check — later removed as redundant: only OWNER/MEMBER/COLLABORATOR can trigger this at all, and a collaborator naming their own branch `claude/implement/*` isn't a privilege escalation, since they already have `contents: write` and could push to `development` directly.)

**First real trigger failed outright, and taught two things:**

1. **Conditional `required` fields in `--json-schema` (`allOf`/`if`/`then`) are not just unconfirmed — they're rejected outright.** The Claude API returned `400: input_schema does not support oneOf, allOf, or anyOf at the top level` and the run failed at turn 0 before Claude did anything. Reverted to the flat schema (a single top-level `required: ["mode", "checks_passed"]`, no per-mode conditionals) — mode-specific field requirements are prose-only in the skill (Step 4), same as they were before that hardening pass.
2. **A hard Claude-step failure left no trace anywhere except the Actions log.** All three outcome-handling steps (create PR / iterate / blocked) require `structured_output` to exist, so when the claude step errors out before producing any, none of them run — nobody watching the issue would know anything happened. Added a failure-notice step, gated on `always() && steps.claude.outcome == 'failure'`, that posts a comment linking to the failed run. Also hardened the other three steps' `if:` conditions with an explicit `steps.claude.outputs.structured_output != ''` check before the `fromJSON(...)` call — an empty string previously produced a second, more confusing "template is not valid" error on top of the real one, since `fromJSON('')` throws.

**Second real trigger ran out of `--max-turns` (200) on an ordinary-sized feature.** First fix: raised to 500 (turns cover every tool call — reading context docs, edits across backend/frontend, and up to three full check-and-fix cycles all add up fast). Initial assumption was that the harness had already committed whatever it got done to a branch before failing, so a "salvage" step was added that checked for commits ahead of `development` on `steps.claude.outputs.branch_name` via `gh api compare` and opened a draft PR if any existed.

**That assumption was wrong.** On the actual failing run, Claude had made substantial edits but no branch existed at all — meaning the harness only finalizes its own commit+push once a session completes *normally*; a run that dies mid-session (hits `--max-turns`, errors) leaves whatever it edited sitting only in the job's local working tree, about to be destroyed with the runner regardless of how much real work was done. The fix generalizes what "salvage" means: a new recovery step runs on any Claude-step failure and commits anything still uncommitted itself (as `claude[bot]`), then pushes `HEAD` — from the harness's own commit, from Claude's, or from the recovery step's own commit, whichever combination actually happened — as long as it differs from a SHA captured right after checkout, before Claude ran. This works regardless of *where* in that chain the harness stopped, rather than depending on an assumption about when it commits. The old salvage step now consumes this recovery step's output instead of independently re-deriving branch state.

**Third real trigger completed *successfully* — checks failed as expected, but PR creation then failed with `gh: Bad credentials (HTTP 401)`, and the branch it tried to compare against was empty.** This forced reading the action's actual source rather than continuing to infer behavior from its logs, and overturned the previous entry's assumption in a more fundamental way: **agent mode (the explicit `prompt:` this workflow uses) never runs the action's branch-setup code at all.** That code — which reads `branch_prefix`/`branch_name_template` and creates a local branch — is tag-mode only. Agent mode instead reads a `CLAUDE_BRANCH` env var directly and falls back to the checked-out ref if it's unset, which is why `steps.claude.outputs.branch_name` had always come back empty and why the PR-creation step's `gh api compare/development...` call, given an empty branch, built a malformed URL. Separately, and more seriously: by default, Claude's *only* commit mechanism in agent mode is raw `git commit`/`git push`, which was never in this workflow's `--allowedTools` — and the alternative, an API-based `commit_files`/`delete_files` MCP tool, only gets installed when `use_commit_signing: true` is set, which it also never was. So in every run up to this point, including this "successful" one, **Claude had no way to commit or push anything at all** — the substantial edits described in its PR body were real, but existed only in the ephemeral runner's working tree and vanished with it. (The earlier max-turns recovery step happened to work anyway, since it runs its own `git` commands directly as a workflow step, outside Claude's tool restrictions — but it only fires on failure, so it never covered a successful run like this one.) Fixed by setting `use_commit_signing: true` and adding `mcp__github_file_ops__commit_files`/`mcp__github_file_ops__delete_files` to the allowlist (Claude's real, intended commit path — still no raw `git` for it), computing the branch name in its own deterministic pre-Claude workflow step and passing it in via `CLAUDE_BRANCH` (so `steps.branch.outputs.name` is the one source of truth every later step uses, instead of trusting the action's echo of it back), and instructing the skill to commit incrementally via `commit_files` plus once more immediately before its final output — directly answering the original "commit more often" ask, now via the mechanism actually built for it instead of a workflow-level guess.

**Fourth real trigger got all the way through — commit, checks, structured output all worked — and failed only on `gh pr create`, with `HTTP 401: Bad credentials`.** Root cause: `steps.claude.outputs.github_token` (the real GitHub App token the action mints via OIDC, used specifically because it — unlike the default `GITHUB_TOKEN` — triggers other `pull_request`-listening workflows) is revoked by the action itself as its own last internal action once Claude's turn ends, confirmed live via a `curl -X DELETE .../installation/token` call visible in the action's own log output, running *before* control ever returns to this workflow's own steps. Every deterministic follow-up step that referenced that token was therefore always going to get a dead token, on every single run, not just this one — the earlier failures had simply never gotten far enough to reach this bug. Fixed by switching the three affected steps (create PR, iterate summary, blocked comment) to `github.token` instead. Trade-off accepted for now: PRs/comments made with the default `GITHUB_TOKEN` don't trigger other workflows (`ci.yml`, the title-lint check) — a deliberate GitHub anti-recursion rule — so a `claude[bot]`-opened PR won't auto-run CI until a real GitHub App/PAT credential is provisioned as a secret to replace `github.token` here specifically. Not done in this pass; flagged as a follow-up.

Same run also surfaced a quieter correctness gap: `php artisan test`/`vendor/bin/phpunit` were denied ("requires approval") every time Claude ran them after `cd`-ing into `api/` first, because the allowlist patterns (`Bash(php api/artisan test:*)`, `Bash(api/vendor/bin/pint:*)`) require the literal `api/`-prefixed invocation form — while the frontend `npm run *` patterns have no path prefix and matched regardless of cwd. The run still reported `checks_passed: true`, with the gap only visible in `checks_summary` prose, not surfaced anywhere more prominent. Fixed by adding the bare (no-`api/`-prefix) forms — `Bash(vendor/bin/pint:*)`, `Bash(php artisan test:*)` — to the allowlist alongside the existing ones, so the check runs regardless of which cwd Claude happens to invoke it from.

**Fifth real trigger got all the way to a genuinely clean run — real commit, all checks actually green including backend tests this time — and `gh pr create` still failed**, now with `GraphQL: GitHub Actions is not permitted to create or approve pull requests`. This is a repo-level guardrail ("Allow GitHub Actions to create and approve pull requests" under Settings → Actions → General), off by default on every repo, that specifically restricts the default `GITHUB_TOKEN` — which is exactly what the previous fix had switched to. Two ways out were considered: enable that repo setting and keep `github.token`, or have Claude create the PR itself using its own still-live GitHub App token before the action revokes it. The second was chosen, because GitHub App installation tokens are documented as exempt from that same restriction *and* don't suffer the "doesn't trigger other workflows" limitation `GITHUB_TOKEN`-authored PRs have — so it fixes both the creation-permission problem and the CI-doesn't-run problem from the earlier changelog entry in one move, without touching any repo settings. The trade-off accepted deliberately: `gh pr create` is now allowlisted for Claude specifically, a narrow, one-command exception to the "Claude has zero GitHub write capability" design — justified because the actual content going into the PR (`pr_title`/`pr_body`) was already fully Claude-authored and trusted verbatim either way; only *which* credential submits it changed. One knock-on consequence: the crash-recovery "salvage" step (opens a draft PR for work recovered after a failed run) has no live Claude token to use at that point — it still has to use `github.token`, so it now hits this same restriction and is expected to fail; hardened to fail gracefully (`gh pr create` result checked, falls back to a comment naming the branch) rather than hard-erroring the job.
